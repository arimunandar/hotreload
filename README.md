# hotreload

Hot reload for iOS Simulator. Edit Swift files, see changes instantly — no Xcode rebuild.

Works with **SwiftUI** and **UIKit**.

```
Save .swift → watcher detects → compile (~0.6s) → inject dylib → UI updates
```

## Install

**Quick install:**

```bash
curl -fsSL https://raw.githubusercontent.com/arimunandar/hotreload/main/install.sh | bash
```

**Or build from source:**

```bash
git clone https://github.com/arimunandar/hotreload.git
cd hotreload
cargo install --path .
```

Verify:

```bash
hotreload --help
```

## Setup

### 1. Add HotReloadKit to your Xcode project

In Xcode: **File → Add Package Dependencies** → paste:

```
https://github.com/arimunandar/HotReloadKit
```

HotReloadKit is safe to keep as a permanent dependency — it compiles to no-ops in Release builds (`#if DEBUG`).

### 2. Initialize your project

```bash
cd your-ios-project
hotreload init
```

This:
- Creates `.hotreload/config.toml` (auto-detected from your `.xcodeproj`)
- Configures Xcode build settings (`-interposable`, `-enable-implicit-dynamic`)
- Installs Claude Code skills for AI-assisted hot reload

### 3. Wire up HotReloadKit in your app

#### SwiftUI

Add `.enableInjection()` on the root view — that's all you need:

```swift
import SwiftUI
import HotReloadKit

@main
struct MyApp: App {
    init() {
        HotReload.configure()
    }

    var body: some Scene {
        WindowGroup {
            ContentView()
                .enableInjection()
        }
    }
}
```

This listens for injection notifications and forces the entire view tree to re-evaluate. You do **not** need to add anything to every view.

If a specific view doesn't update after injection (rare — happens when SwiftUI's diffing skips it because its inputs haven't changed), add `@ObserveInjection` to that view only:

```swift
struct StubbornView: View {
    @ObserveInjection var redraw  // only needed as escape hatch

    var body: some View {
        Text("Now I'll refresh on injection")
    }
}
```

#### UIKit

One line in your `AppDelegate` — all view controllers are covered automatically:

```swift
import HotReloadKit

func application(_ application: UIApplication,
                 didFinishLaunchingWithOptions launchOptions: [UIApplication.LaunchOptionsKey: Any]?) -> Bool {
    UIApplication.enableHotReload()
    return true
}
```

This swizzles `viewDidLoad` so every view controller gets injection support. On injection, each visible view controller re-calls `loadView()` → `viewDidLoad()`, so your setup code re-runs with the new implementation. No per-controller setup needed.

> **UIKit tip:** Keep all visual setup in `viewDidLoad()` (or a method called from it). Code in `init` or `awakeFromNib` won't re-run on injection.

### 4. Build, run, and watch

Build your app in Xcode (Debug, iOS Simulator) and launch it. Then:

```bash
hotreload watch
```

Edit any `.swift` file and save. The change appears in the running app in under a second.

## CLI Commands

| Command | Description |
|---------|-------------|
| `hotreload init` | Detect Xcode project, configure build settings, install Claude skills |
| `hotreload watch` | Watch files, auto-compile and inject on save |
| `hotreload inject <file>` | One-shot compile and inject a specific file |
| `hotreload status` | Ping the app to check connection |
| `hotreload log` | Show app-side injection logs (dlopen, TX patches) |
| `hotreload log --stream` | Live stream injection logs |

### Common options

```
-p, --path <dir>   Project root directory (default: current directory)
--app-host <ip>    App's IP address (default: 127.0.0.1)
--app-port <port>  App's TCP port (default: 8899)
--http-port <port> Dylib HTTP server port (default: 9876)
```

### Examples

```bash
# Watch from a parent directory
hotreload watch --path MyApp/

# One-shot inject a specific file
hotreload inject Sources/Views/ProfileView.swift

# Check the last 5 minutes of injection logs
hotreload log --last 5m

# Live tail the logs while iterating
hotreload log --stream
```

## What works / what doesn't

### Works (no rebuild needed)

- SwiftUI `var body` changes — layout, modifiers, colors, text, images
- Computed properties on structs and enums
- Free functions and static methods
- Extension methods on structs/enums

### Requires full Xcode rebuild

- Adding/removing stored properties
- Changes to class definitions (`ObservableObject`, `UIViewController`)
- Protocol or protocol conformance changes
- Adding new types referenced by existing code
- Changes to `@main` app entry point

## Configuration

`.hotreload/config.toml` (auto-generated by `hotreload init`):

```toml
[project]
project_path = "MyApp.xcodeproj"
scheme = "MyApp"
module_name = "MyApp"
target = "arm64-apple-ios-simulator"
sdk_path = "/Applications/Xcode.app/.../iPhoneSimulator.sdk"

[injection]
http_port = 9876
app_port = 8899
app_host = "127.0.0.1"
watch_paths = ["Sources"]
watch_extensions = ["swift"]
debounce_ms = 150
```

Key fields:
- **`module_name`** — must match your app's Swift module name (usually the target name). The dylib's symbols need the same mangled prefix.
- **`watch_paths`** — directories to monitor for changes.

## Claude Code Integration

`hotreload init` installs two Claude Code skills:

**`/hotreload`** — Tell Claude to make UI changes and it will:
1. Check app connection (`hotreload status`)
2. Edit the Swift file
3. Wait for the watcher to compile and inject
4. Verify via `hotreload log`
5. Take a screenshot to confirm

**Verifier** — After any injection, checks dlopen success, TX patch count, and visual confirmation. Reports structured PASS/FAIL.

Example: *"change the banner gradient to orange and pink"* — Claude edits the file, the watcher injects it, Claude verifies the log and takes a screenshot.

## Troubleshooting

### "Could not connect to 127.0.0.1:8899"

The app isn't running or HotReloadKit isn't configured. Build and launch the app in the Simulator first, then check `hotreload status`.

### Compile errors in watcher output

Fix the Swift error in your source file and save again. The watcher will retry automatically.

### "0 TX patches" or changes not visible

- Ensure `OTHER_SWIFT_FLAGS` includes `-Xfrontend -enable-implicit-dynamic` in Xcode Build Settings (or re-run `hotreload init`).
- The changed function may be in a class (ObservableObject). Class method patching is excluded to prevent crashes — these changes need a full rebuild.
- Try navigating away and back, or toggle a state value to force SwiftUI to re-render.

### "dlopen FAILED: symbol not found"

The dylib references a type that isn't exported by the main binary. This happens with struct/enum types defined in files not compiled into the dylib. The watcher compiles all app files together to avoid this.

## How It Works

1. **Compile** — `swift-frontend` compiles each app source file in parallel using `-primary-file` (codegen) while parsing all other files for type-checking. Pre-built `.swiftmodule` files (e.g. HotReloadKit from SPM) are discovered automatically from DerivedData.

2. **Link** — Object files are linked into a `.dylib` with `-undefined dynamic_lookup` (symbols resolved at runtime) and `-interposable`.

3. **Serve** — The CLI starts an HTTP server to serve the dylib file.

4. **Inject** — A TCP command tells the running app to fetch and load the dylib.

5. **Patch** — `InjectionLoader` calls `dlopen()`, scans the main binary's Mach-O symbol table for TX (dynamic replacement) variables, finds matching functions in the dylib via `dlsym`, and writes new function pointers.

6. **Re-render** — A notification triggers SwiftUI to re-evaluate view bodies with the patched implementations.

```
hotreload (Rust CLI)              HotReloadKit (iOS Swift Package)
====================              ================================

File Watcher (notify)             TCP Server (BSD sockets)
       |                                 |
Compiler (swift-frontend)         HTTP Client (URLSession)
  parallel -primary-file                 |
  DerivedData module discovery    InjectionLoader
       |                           dlopen + TX patching
HTTP Server (Warp)                       |
       |                          SwiftUI re-render
TCP Injector  ──── reload ────>   Process command
              <─── ok ────────
```

## Limitations

- **iOS Simulator only** — physical devices block `dlopen()` of unsigned code
- **Debug builds only** — release optimizations inline functions, preventing dynamic replacement
- **Class methods excluded** — patching `ObservableObject`/`UIViewController` methods causes crashes due to ObjC class duplication; these require a full rebuild
- **`#Preview` macros** — incremental compilation skips `#Preview` blocks (macro plugins unavailable to `swift-frontend`)

## Requirements

- macOS with Xcode 16+ (Swift 6.x)
- Rust toolchain (only if building from source)
- iOS Simulator
